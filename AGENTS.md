# AI 開発協働ガイド

## Definition of Done (DoD)

すべてのタスク完了時に以下を満たすこと。

### 1. 品質チェック（必須）

- 型エラー 0 件
- リント違反 0 件
- 全テスト成功
- フォーマッター適用済み

### 2. スコープ遵守

- 指示された範囲のみ変更する
- 余計な機能追加・リファクタリング・ドキュメント生成を行わない
- 既存ファイルの編集を優先し、新規ファイル作成は必要最小限

### 3. 再現可能性

- 変更後のビルド・テストが第三者環境でも成功すること
- 環境依存の暗黙知を残さない

## Failure Policy

同一アプローチで **3回失敗** したら、それ以上リトライせず以下を報告する:

1. 試行内容: 何を3回試みたか
2. 失敗理由: 各回の具体的エラー
3. 代替案: 別のアプローチの提案（あれば）

## 実装ワークフロー

1. 計画 - 大規模変更時は事前に変更概要を共有し承認を得る
2. コード理解 - ツール優先順位に従い効率的に探索
3. 実装 - 型安全・影響範囲確認
4. 品質保証 - DoD の全項目を実行
5. 報告 - 完了報告、問題があれば早期相談

## 重要な開発原則

- **型安全性**: any 型・型アサーション排除
- **エラーハンドリング**: 適切なエラー処理パターン
- **未使用変数**: 原則削除。例外的に `_` プレフィックス許可（catch 文、フレームワーク必須引数のみ）

## コード探索の原則

**優先順位**（トークン効率重視）:

1. MCP Serena - シンボル解析・依存関係追跡
2. Grep - パターン検索・横断的調査
3. Read - 設定ファイル・小規模ファイル・最終手段

## ファイル操作原則

- 既存ファイル編集を優先、新規作成は必要最小限
- README.md や \*.md は明示的に要求された場合のみ作成
- 既存テストなど重要なものは勝手に削除しない

## ドキュメント作成の優先順位

新しい知識を追加するとき:

1. Skill - 繰り返し使う知識
2. Agent - 自動実行すべきタスク
3. Command - ユーザーが手動実行する操作
4. Rules/Steering - プロジェクト固有のルール
5. Docs - 上記で表現できない場合のみ、最小限

## Git コミット規約

- 変更内容を簡潔に記述するのみ
- 追加の署名やフッターは不要

## コミュニケーション原則

- 分からないことは素直に報告し、思考プロセスをオープンにする
- 内部で英語思考してもよいが、最終回答は必ず日本語
- 不確実性・問題発見時は早期相談

## 禁止事項

- `find`, `grep`, `cat`, `head`, `tail` の直接使用（専用ツールを使うこと）
- 要求されていない機能追加・リファクタリング
- 既存テスト・重要ファイルの無断削除
- any 型の導入
